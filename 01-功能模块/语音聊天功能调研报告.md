# 语音聊天功能调研报告

**报告版本**: v3.0  
**生成日期**: 2026-04-29  
**最后更新**: 2026-04-29  
**审查范围**: `src/game/client/components/qmclient/voice_*`、`src/game/client/components/qmclient/qmclient.cpp`、`src/test/voice_core_test.cpp`  
**更新说明**: 本报告用于替换旧版以 Rust 语音模块为核心的调研结论。当前 `main` 上的语音实现已经回到纯 C++ / `QmVoice*` 配置体系，旧版 Rust 覆盖率、Rust 架构优劣、`RiVoiceUseRust` 切换等结论不再适合作为现状文档。

## 一、执行摘要

### 1.1 当前结论

1. 当前语音模块是 **纯 C++ 实现**，核心文件为 `voice_component.*`、`voice_core.*`、`voice_utils.*`，没有现行 Rust 运行路径。
2. 模块整体功能仍然完整覆盖了语音聊天的核心链路：设备初始化、网络收发、Opus 编解码、VAD/PTT、空间音频、房间 Token、可见性/名单过滤、说话叠加层。
3. 当前实现的主要风险不在“功能缺失”，而在 **线程安全、平台音频设备行为、协议一致性、端到端验证不足**。
4. 旧版文档中关于 `src/rust-bridge/voice/`、Rust 超越 C++、`RiVoiceUseRust` 双模切换等内容，已经不代表当前分支现状。

### 1.2 当前状态概览

| 维度 | 结论 |
|------|------|
| 运行架构 | 纯 C++ |
| 核心协议 | 自定义 `RV01` UDP 协议 |
| 音频 I/O | SDL2 音频设备 |
| 编解码 | Opus |
| 降噪 | 三态模式：不降噪 / 简单降噪 / RNNoise 降噪（当前开发构建已默认集成 RNNoise，运行时可直接切换） |
| 控制方式 | VAD / PTT / 测试模式 |
| 识别同步 | 通过 `qmclient.cpp` 的 `/qm/*` HTTP 接口做 Qm 标识、在线分布和语音支持标记 |
| 自动化测试 | 以 `voice_utils` 单元测试为主，缺少端到端语音测试 |

### 1.3 降噪模式表述修正

旧文档里常见的「RNNoise（编译启用时）」是一个历史性的简写。现在这层歧义已经被消掉了：RNNoise 源码和官方模型文件已经直接 vendoring 到客户端构建里，开发者正常构建时默认就会带上这条能力。

旧写法的问题在于把**构建能力**和**用户运行时选择**混在了一起，读者容易误以为“只要编进去了，客户端就固定总是使用 RNNoise”。

现在更准确的表述应该是：

- 当前开发构建已经默认集成 RNNoise，运行时可以直接切到 RNNoise 路径。
- 客户端运行时提供三态降噪模式：`0=不降噪`、`1=简单降噪`、`2=RNNoise 降噪`。
- 默认语义仍然是「简单降噪」；是否切到 RNNoise，由用户在设置页自行选择。
- 代码仍然保留“未集成时自动回退到简单降噪”的保护分支，用于未来裁剪构建或异常产物，但不再是日常开发构建的常态。

## 二、旧报告已过时的部分

下面这些点在旧版《语音聊天功能调研报告》中已经不再准确：

1. **审查范围错误**  
   旧报告以 `src/rust-bridge/voice/` 为主，而当前语音主实现不在该目录。

2. **“Rust 100% 覆盖 C++ 功能”结论过时**  
   当前现状不是 Rust 接管运行，而是纯 C++ 链路。

3. **`RiVoiceUseRust` / 双模式切换结论过时**  
   当前 `voice_component.h` 只有 `CRClientVoice m_Voice;`，没有 Rust 组件成员，也没有运行时切换接口。

4. **配置命名体系已变化**  
   当前配置前缀是 `QmVoice*` / `qm_voice_*`，而不是旧报告中的 `RiVoice*` 主导。

5. **旧版功能覆盖率统计不可直接复用**  
   旧报告统计的是 Rust 相对 C++ 的覆盖程度，不是当前 C++ 模块本身的完备性。

6. **“Rust 明显优于 C++”类结论不再是现状文档应有内容**  
   当前维护对象是 C++ 模块，应该描述其真实行为、风险和边界，而不是继续围绕 Rust 做优劣对比。

7. **旧版 Windows Opus DLL 修复附录不再是当前语音现状报告重点**  
   它属于特定构建环境问题，和当前功能梳理不是同一层面。

8. **旧版提到的某些 Rust 新增能力（AEC、AGC2、MOS）不能再默认视为现有功能**  
   当前 C++ 现状报告应以 `voice_core.cpp`、`voice_utils.cpp` 的真实代码为准。

## 三、当前模块结构与职责

### 3.1 文件边界

| 文件 | 主要职责 |
|------|----------|
| `src/game/client/components/qmclient/voice_component.h/.cpp` | 组件入口、控制台命令注册、OnInit/OnRender/OnShutdown、叠加层导出 |
| `src/game/client/components/qmclient/voice_core.h/.cpp` | 核心语音运行时：设备管理、工作线程、协议收发、Opus、VAD/PTT、空间音频、房间成员状态 |
| `src/game/client/components/qmclient/voice_utils.h/.cpp` | 协议读写辅助、音频数学工具、名单匹配、增益/压缩/3D 声场计算 |
| `src/game/client/components/qmclient/voice_callback.h` | SDL 音频回调相关接口声明 |
| `src/game/client/components/qmclient/qmclient.cpp` | Qm 识别、在线分布、语音支持标记同步（非语音 PCM/UDP 本体） |
| `src/test/voice_core_test.cpp` | `voice_utils` 相关单元测试 |

### 3.2 组件入口

当前组件入口非常直接：

- `CVoiceComponent::OnInit()` 调用 `m_Voice.Init(...)`
- `CVoiceComponent::OnRender()` 在 `qm_voice_enable` 开启时调用 `m_Voice.OnFrame()`
- `CVoiceComponent::RenderOverlay()` 使用 `CVoiceOverlayState` 渲染左侧语音说话列表

这说明当前没有 Rust/C++ 混跑调度层，UI 和组件层只面向 `CRClientVoice`。

## 四、运行时线程模型

### 4.1 主线程

主线程职责主要有：

1. 读取配置和 UI 输入
2. 维护客户端快照（本地 ID、玩家位置、名字、观战状态）
3. 调用 `OnFrame()` 驱动语音系统生命周期
4. 导出叠加层状态给 UI

### 4.2 Worker 线程

`CRClientVoice` 内部维护独立工作线程，核心循环包括：

1. `EnsureAudio()`
2. `ResolveServerAddr()`
3. `ProcessIncoming()`
4. `DecodeJitter()`
5. `UpdateEncoderParams()`
6. `ProcessCapture()`

Worker 线程同时负责：

- 设备重试
- UDP 收发
- 抖动缓冲处理
- Opus 编解码
- PING/保活
- 发话状态更新

### 4.3 SDL 音频回调线程

输出设备通过 SDL 音频回调拉取混音结果，`MixAudio(...)` 会从各 peer 已解码帧中取 PCM 做混音。

### 4.4 线程模型风险

当前实现把主线程、worker 线程和 SDL 回调线程混合在一起运行，这也是该模块风险最集中的原因之一：共享状态多、边界复杂、回归不易靠单元测试覆盖。

## 五、协议与收发流程

### 5.1 协议头

当前语音协议仍使用 `RV01` 魔数，自定义 UDP 包头大致包含：

1. `VOICE_MAGIC`
2. `version`
3. `type` (`AUDIO` / `PING` / `PONG`)
4. `payload_size`
5. `context_hash`
6. `token_hash`
7. `flags`
8. `sender_id`
9. `sequence`
10. `pos_x`
11. `pos_y`

对应读写辅助在 `voice_utils.cpp` 中统一实现。

### 5.2 发送链路

发送侧核心流程：

1. 从主线程快照同步本地位置、本地 client id、观战位置等上下文
2. 根据配置决定 VAD / PTT / 测试模式
3. 在需要时发送 `PING` / keepalive
4. 从 capture 设备取 PCM
5. 做麦克风增益、降噪、高通压缩等音频质量优化
6. 根据 VAD/PTT 判断是否发话
7. Opus 编码
8. 封包并通过 UDP 发送

这里的「音频质量优化」当前至少包含三类能力：

1. **输入增益**：通过麦克风音量配置调节输入电平。
2. **降噪模式**：支持不降噪、简单降噪、RNNoise 降噪三种模式。
3. **动态处理**：高通、压缩、限幅等链路用于尽量压住底噪和爆音。

### 5.3 接收链路

接收侧核心流程：

1. 校验来源地址必须等于当前语音服务器地址
2. 校验包头魔数、协议版本和包类型
3. 校验 `context_hash`
4. 按 `token_hash` / group 规则判断是否可听
5. 对音频包进入 peer jitter 队列
6. 在 `DecodeJitter()` 中解码为 PCM 帧
7. SDL 输出回调线程混音播放

### 5.4 房间与识别同步不是同一条链

需要特别说明：

- **UDP 语音聊天本体**：`voice_core.cpp`
- **Qm 标识 / 在线分布 / voice_supported 标记**：`qmclient.cpp`

二者相关但不相同。用户层面会一起感知为“语音系统”，但调试时必须分开看。

### 6.4 降噪模式与默认语义

当前 `qm_voice_noise_suppress_enable` 不再应被理解为简单的布尔开关，而应理解为一个三态模式值：

| 值 | 模式 | 说明 |
|----|------|------|
| `0` | 不降噪 | 保留原始输入，不做降噪处理。 |
| `1` | 简单降噪 | 使用基础降噪路径，作为默认语义。 |
| `2` | RNNoise 降噪 | 显式选择 RNNoise 路径；当前开发构建默认可用，并保留异常情况下的回退保护。 |

`qm_voice_noise_suppress_strength` 保持为现有强度控制项，但 UI 文案应跟随当前模式显示，例如「简单降噪强度」或「RNNoise 降噪强度」。

## 七、Qm 识别与在线分布同步

### 7.1 当前职责

`qmclient.cpp` 中的识别同步主要用于：

1. 获取 `Qm` 识别 token
2. 上报本地玩家与语音支持状态
3. 拉取 `users.json`
4. 在本服玩家上打 `Qm` 标识和 `voice_supported` 标记
5. 生成“栖梦客户端在线分布”面板数据

### 7.2 路由分工

当前代码和线上行为体现出两个 HTTP 服务面：

1. **默认语音服务（默认 `:9987`）**
   - `Qm` 识别
   - 在线分布
   - 语音支持标记
   - 路径使用 `/qm/*`

2. **中心 HTTP 服务（`:8080`）**
   - 版本
   - 健康检查
   - 在线时长
   - 路径不使用 `/qm/*`

这部分很容易被误改，因此建议持续保留源码注释说明。

当前测试已经从“工具函数正确性检查”扩展到“协议决策 + 接收可听性规则 + 设备诊断 + 设备枚举/断开 UI 状态机 + 权限提示链路 + Qm 同步解析”的组合保障，但仍不是完整的语音可用性闭环。

## 九、当前风险、现象与修复方向

### 9.1 高风险：多线程共享状态复杂（已修复）

**当前状态：已修复当前确认的高风险线程边界问题。**

本轮已经完成的内容：

1. 主线程的重建设备判断已经收敛为 `VoiceNeedsAudioRefresh(...)` 这类纯快照决策，不再为了声道变化之类的条件直接在 `OnRender()` 里关闭音频设备。
2. 设备、编码器、Socket 的关键“就绪”状态已经增加原子化快照位，主线程更多只读这些快照并发出重建请求，真正的设备生命周期继续交给 worker 线程处理。
3. worker 线程中的 `EnsureAudio()` 已不再直接读取 live `g_Config`。音频后端、输入设备、输出设备、立体声开关现在统一由主线程锁存到一份请求态，再由 worker 消费同一份请求态，避免“主线程判定一套、worker 实际应用另一套”的跨线程竞态。
4. `SetPttActive()` 的 PTT 释放延迟在读配置前会先强制刷新 snapshot，避免把刚修改的参数滞后一个快照周期才生效。
5. 源码里已经补充“主线程写请求态、worker 只消费快照”的注释，后续再改音频设备切换逻辑时不需要靠口头约定维持边界。
6. UI 状态面板读取的音频/网络/编解码诊断字符串，已经改成通过单独锁做快照导出，不再让 worker 写字符串、主线程读字符串之间裸奔竞争。

`CRClientVoice` 在主线程、worker 线程、SDL 回调线程之间共享大量状态，例如：

- peer 队列
- 音频设备句柄
- 说话状态
- 快照缓存
- 输出混音相关状态

这也是历史代码审查文档中反复提到的最大风险面。当前没有继续发现新的 blocker 级线程边界漏洞，高风险结论可以收口。

**实际影响：**

- 偶发音频状态不同步
- 设备切换后残留旧状态
- 某些平台上出现“UI 显示已开启，但内部链路未真正就绪”

**后续可继续优化：**

1. 把采集、播放、peer 状态拆成更清晰的状态块，继续降低维护复杂度。
2. 进一步把设备句柄、输入输出启停状态整理成更统一的状态机，减少布尔组合阅读成本。
3. 针对设备切换、断开重连、退出房间补一轮状态回收检查，作为中风险稳健性增强项继续推进。

### 9.2 高风险：协议一致性回归难以及时发现（已修复）

**当前状态：已修复。**

本轮已经完成的内容：

1. `voice_utils` 的协议读写、模式解析、发送前置条件、编码参数等逻辑补齐了单元测试。
2. `qm_voice_noise_suppress_enable` 已从布尔语义收敛为明确的三态模式，减少了配置值和处理路径不一致的风险。
3. RNNoise 的运行时选择、异常回退和基础 API 可调用性已经进入测试覆盖，不再是“编译时看起来有、运行时不确定”的状态。
4. `RV01` 包头已经补了固定测试向量，覆盖 `AUDIO` / `PING` / `PONG` 三类包，降低“改了字段顺序但测试没发现”的风险。
5. 协议字段用途已经补到 `SVoicePacketHeader` 附近的源码注释里，后续改包不再只能靠记忆。
6. 调试日志中的发送前置条件已经统一折叠为 `tx_block` 位掩码，同时也会输出可读的 `tx_blockers=...` 文案。
7. 接收侧过滤原因已经拆成更细的计数：地址、头部、版本、类型、上下文、分组、sender、payload、距离等，不再只有 `drop_ctx` / `drop_radius` 两个粗粒度结果。
8. `context_hash`、`token_hash`、`sender_id`、`sequence` 的读写一致性现在有独立测试，不再只靠一个大而全的 round-trip 测试隐式覆盖。
9. `ProcessIncoming()` 的关键过滤决策已经抽成可测 helper，版本、类型、上下文、分组、sender、payload、keepalive token 以及 `PING/PONG` RTT 前置条件都已经有针对性测试。

语音协议是手写二进制协议，包头字段较多，任何一个字段写错、偏移错位、配置快照与读包逻辑不一致，都可能表现为：

- `ping = -1`
- `rx = 0`
- 看不到房间内其他说话者
- 听不到别人

这类问题如果没有协议级测试，很容易靠运行时现象才暴露。

**实际影响：**

- 用户第一感知是“语音坏了”，但日志表面上只看到 `peers=0`、`rx=0`
- 一处字段改动可能同时影响房间识别、上下文过滤和收发联通

**后续可继续优化：**

1. 增加一组更贴近真实收包路径的最小正向冒烟测试，补齐“成功进入 jitter/解码链路”的自动化证据。
2. 如果后续继续扩展协议字段，保持固定向量测试与 `ProcessIncoming()` 分类测试同步更新。

### 9.3 中风险：设备枚举与平台差异（已修复代码层风险）

**当前状态：代码层风险已修复，剩余主要是外部平台行为验证。**

本轮已经完成的内容：

1. 发送链路前置条件被显式整理为 `VoiceTransmitBlockers(...)`，至少能更稳定地区分网络、采集、编码器和静音导致的“发不出去”。
2. 菜单中的降噪模式、降噪强度文案和回退提示做了收敛，减少了“设置看起来开了，实际不是那条链路”的误解。
3. 调试日志里已经能直接看到更细的收发阻塞与过滤原因，这对 macOS / 耳机 / 默认设备问题的初步排查帮助很大。
4. 设置页已经新增一组面向用户的状态字段，直接展示“麦克风 / 扬声器 / 服务器 / 房间 / 收发”当前卡在哪一步，不再只能靠 `cap=0`、`rx=0` 这类开发日志硬猜。
5. 设备错误已经补了一层可测试的诊断归类：输入/输出设备不存在、设备打开失败、权限拒绝、后端初始化失败都能稳定映射到固定 hint，不再只能靠字符串全文匹配人工判断。
6. 输入/输出设备下拉的默认项、大小写去重、当前已选设备掉线保留、断开回显逻辑已经抽成纯 helper 并进入测试，至少能更早拦住“热插拔后 UI 把当前设备选项吞掉”这类回归。
7. `VoiceUiActionHint(...)` 已经开始优先消费更细粒度的音频失败 hint；像 macOS 麦克风权限拒绝、设备不存在、设备打开失败这类问题，设置页不再只给出一条笼统的“检查输入设备”建议。

本轮继续完成的内容：

8. `ExportUiStatus(...)` 现在已经把“请求的输入/输出设备”“当前已解析/已打开的设备”“是否正在等待音频重建”导出到统一 UI 状态结构。
9. 设置页统一状态面板已经新增“输入切换”“输出切换”“音频问题”三类结果字段，能直接告诉用户当前是在切换中、已经切到指定设备、默认设备打开失败、指定设备打开失败，还是麦克风权限被系统拒绝。
10. 设备初始化失败原因已经系统性抬升到统一面板，不再只藏在底层日志里。
11. 设备下拉切换后已经具备“正在切换 / 已切换到 / 切换失败”的即时反馈，用户不再只能靠猜。

当前模块高度依赖 SDL 设备枚举和设备名匹配：

- 设备名大小写
- 已断开设备回显
- 默认设备解析
- 平台权限 / 后端差异

尤其在 macOS 上，输入设备不可用、系统权限、默认设备行为都可能直接表现为 `cap=0`。

**实际影响：**

- 用户会看到语音开关已经打开，但自己听不到、别人也听不到
- 有线耳机、蓝牙耳机、系统默认设备切换后，行为可能和 UI 预期不一致

**剩余外部验证项：**

1. macOS 默认输入/输出路由的真实切换行为仍需要真机验证，因为 SDL 和系统音频路由是运行环境外部行为。
2. 有线耳机、蓝牙耳机、系统默认设备漂移仍需要真插拔/真切换验证。
3. 平台权限弹窗本身仍属于系统交互，不属于客户端内部状态机可完全替代的部分。

### 9.4 中风险：Qm 识别同步与语音服务地址耦合（已修复）

**当前状态：已修复。**

识别同步 URL 不是固定的中心域名，而是从当前语音服务器地址推导出来。这样做有复用性，但也意味着：

- 改动语音服务器时会一起影响 Qm 标识和在线分布
- 路由前缀和端口职责容易被误改

**实际影响：**

- 一次看似只改“显示标识”的调整，可能顺带把语音配套识别链路改坏
- `/qm/*` 路由与 `:8080` 中心服务职责混淆时，最容易出现“Qm 标识没了”和“语音也异常”的连锁误判

**修复方向：**

1. 在 `qmclient.cpp` 中保留路径职责注释，明确 `:9987` 与 `:8080` 的分工。
2. 将 Qm 标识同步、在线分布同步、语音联通状态打印拆成独立日志项。
3. 补一条联调说明：改动 `/qm/*` 前必须同时验证标识、`users.json`、语音支持标记。

本轮已经完成的内容：

1. `users.json` 解析规则已经抽成纯 helper，`server_address|server`、`player_id|id`、`voice_supported`、`dummy`、本服 marks 聚合和排序规则都有单元测试兜底。
2. `:9987` 的 `/qm/*` 路径职责注释已经补在源码常量附近，后续再改路由时更不容易把语音服务和 `:8080` 中心服务混写。
3. 这意味着以后再动 `/qm/users.json` 或 `/qm/token` 时，至少能先在自动化里发现“标识解析和语音支持标记回退被改坏”。
4. Qm 识别 token、在线分布拉取/解析、语音支持上报现在已经拆成独立日志项，后续再出问题时，可以直接区分是 recognition、distribution 还是 voice_presence 哪条链路异常。
5. `users.json` 拉取请求现在也已经归到 distribution 日志，而不是混在 recognition 里。
6. 语音支持上报的 `report_ok/report_failed` 现在会携带真实的 server address / player count 上下文，并且只在 2xx 响应时记为成功，避免日志把失败请求误写成成功。

**后续维护要求：**

1. 改 `/qm/token` 时看 `recognition` 日志。
2. 改 `/qm/users.json` 时看 `distribution` 日志。
3. 改 `/qm/report` 或 `voice_supported` 上报时看 `voice_presence` 日志。

### 9.5 中风险：端到端验证不足（已收敛为外部联调项）

**当前状态：客户端内部可自动化部分已修复，剩余为外部联调验证项。**

本轮已经完成的内容：

1. `voice_utils` 相关测试显著扩展，覆盖了协议头读写、前置条件判定、降噪模式解析、编码参数和音频处理辅助逻辑。
2. RNNoise 现在不仅默认编进开发构建，还补了基础 API 创建与处理链路测试，能证明“库和模型能被真实调用”。
3. 额外做了 x86_64 方向的最小编译校验，避免 RNNoise 仅在当前 arm64/macOS 环境下成立。
4. 新增了一组基于真实包字节的正向协议冒烟测试，以及 `users.json` 解析联调测试，自动化覆盖面已经从 `voice_utils` 扩到语音协议入口和 Qm 同步入口。
5. 设备枚举 / 下拉状态机现在已经有独立自动化测试，覆盖默认设备、大小写去重、当前已选设备掉线后保留与断开回显。
6. 权限拒绝、输入/输出设备不存在、设备打开失败等错误现在已经打通到统一 action hint，并有自动化测试覆盖“从错误字符串到排障建议”的完整映射。
7. “已匹配到房间但仍未收到回声”这一类常见的同房间联通诊断路径，现在也有了独立测试，不再只靠 UI 肉眼点点看。

但需要明确：这些并不等价于“系统外部条件已经自动化”。剩余未自动化的内容主要是：

1. 真实双客户端、真实语音服务、真实媒体面互听。
2. 真设备热插拔、默认设备路由漂移、系统权限弹窗。
3. 系统设置交互和用户授权动作本身。

这三类内容已经从“代码层风险”收敛成“外部联调验证项”。也就是说：客户端内部状态机、协议、设备诊断、房间匹配提示、权限拒绝提示、设备切换反馈、Qm 同步解析和职责拆分已经进入自动化或可观测性保护；剩下的不是未修复 bug，而是任何桌面音频应用都仍需面对的运行环境联调。

### 9.6 剩余优先级说明

当前报告里原先列出的代码层高风险和中风险项，已经收敛为两类：

1. **已修复并进入自动化 / 可观测性保护的代码项**  
   协议字段一致性、收发前置条件、设备下拉状态机、权限/设备错误提示、设备切换反馈、运行态 reset、Qm `users.json` 解析、Qm 同步职责注释与日志拆分，已经不再作为“待修复风险”保留。
2. **仍需继续做的外部验证项**  
   真实双端互听、真设备热插拔 / 默认路由漂移、系统权限弹窗与系统设置交互，仍然需要集成或真机联调验证。

也就是说，从客户端代码层面看，本报告原先列出的风险已经完成修复；剩余工作重点不再是继续补丁，而是把外部环境验证流程沉淀下来。

## 十、功能现状与用户体验评估

### 10.1 当前功能覆盖

从最终用户角度看，当前语音模块已经具备一套相对完整的使用闭环：

1. 可以在设置页启用语音并配置服务器、Token、输入输出设备。
2. 可以在 VAD、PTT、本地测试模式之间切换。
3. 可以看到左侧说话叠加层，感知当前房间内谁在说话。
4. 可以通过 Qm 标识和在线分布感知对方是否可能支持当前语音体系。
5. 可以按距离、全图收听、群组等规则控制可听范围。
6. 可以按需求在不降噪、简单降噪、RNNoise 降噪之间切换，并继续复用统一的降噪强度配置。
7. RNNoise 已经不是“占位模式”：当前构建会真实编入官方 RNNoise 源码与模型，现有测试已经覆盖基础 API 的创建与处理链路。

### 10.2 当前用户体验优点

当前体验里，真正有价值的点主要有：

1. **配置入口集中**  
   语音相关设置已经集中在 QmClient 的配置界面里，学习成本不高。
2. **房间和身份感知较强**  
   不是单纯“开麦就发”，而是把房间 Token、Qm 标识、在线分布整合进来了。
3. **玩法适配度高**  
   支持空间音频、观战位置、距离衰减，这些都更贴近 DDNet/QmClient 的实际使用场景。
4. **调试信息基础已存在**  
   虽然还不够友好，但 `voice:` 日志与状态字段已经为继续打磨提供了基础。
5. **输入音频优化链路完整度较高**
   麦克风增益、降噪模式、高通/压缩/限幅这类音频质量优化已经进入主链路，不再只是外围实验参数。

### 10.3 当前用户体验痛点

从这轮排查过程来看，当前体验上的主要痛点不是“功能没有”，而是“用户不知道系统卡在哪一步”：

1. 开启语音后，用户很难分辨是没拿到麦克风、没连上服务端、没进同房间，还是被协议过滤了。
2. 左侧说话列表、Qm 标识、实际语音联通不是同一个子系统，用户体感上却会把它们视为一件事。
3. 设备切换后的反馈不够直接，尤其在 macOS 和耳机设备场景下，容易误判。
4. 调试字段更偏开发视角，例如 `cap=0`、`rx=0`，普通用户很难据此自助判断问题。
5. 用户仍然不一定理解「简单降噪」和「RNNoise 降噪」的差别，尤其不知道何时该切换模式。

### 10.4 后续用户体验提升方向

如果目标是提升“能不能顺利用起来”，建议优先做下面这些改进：

1. **把状态拆给用户看**  
   在 UI 中明确显示：
   - 麦克风：已打开 / 未打开
   - 扬声器：已打开 / 未打开
   - 服务器：已连接 / 未连接
   - 房间：已匹配 / 未匹配
   - 收发：正在发送 / 正在接收 / 暂无对端
2. **把错误翻译成人话**  
   例如把 `cap=0` 转成“未打开输入设备或无麦克风权限”，把 `peers=0` 转成“当前未发现可通话对端”。
3. **增加一键自检流程**  
   点击后依次检查权限、设备、服务器、Token、收发状态，并输出结论。
4. **优化设备选择反馈**  
   用户选择输入输出设备后，立即提示“已切换成功”或“切换失败，原因是……”
5. **增加首次使用引导**  
   首次开启语音时提示：需要同服、同房间、输入设备正常、对方也支持语音。
6. **把降噪模式解释成用户能理解的话**
   明确告诉用户「简单降噪」适合默认使用，「RNNoise 降噪」适合在背景噪声更重时手动切换尝试。
7. **补充构建与体积说明**
   RNNoise 默认集成后，仓库和目标产物体积会增加，后续如果要继续优化发布体积，需要单独评估是否切换更小模型或改为资源打包方案。

### 10.5 后续工程改进方向

如果目标是同时提升“可维护性”和“用户体验”，建议把后续工作拆成三条线：

1. **可观测性线**  
   完善日志、状态面板、错误提示，让问题更快暴露。
2. **稳定性线**  
   收敛线程边界、补协议测试、补设备异常路径处理；其中换服/掉线/换房间 token 的运行态 reset 已经开始统一。
3. **体验线**  
   优化设置页文案、自检流程、设备切换反馈、首次使用引导。

## 十二、最终结论

当前 QmClient 语音模块已经不是旧报告描述的 Rust 主导实现，而是一个 **纯 C++、多线程、SDL+Opus+自定义 UDP 协议** 的完整语音子系统。

它的优势是：

1. 功能面完整
2. 配置项齐全
3. UI/控制台入口成型
4. 已集成 Qm 识别、在线分布与语音支持标记

它的主要问题不是“没有功能”，而是：

1. 线程安全风险高
2. 平台/设备行为复杂
3. 协议回归容易漏检
4. 端到端自动化验证不足

因此，当前更适合把它定义为：

**“功能完整、玩法适配度高，但在可观测性、线程边界和用户自助排障体验上仍需继续打磨的生产模块。”**
