# 语音聊天功能调研报告

**报告版本**: v3.0  
**生成日期**: 2026-04-29  
**最后更新**: 2026-04-29  
**审查范围**: `src/game/client/components/qmclient/voice_*`、`src/game/client/components/qmclient/qmclient.cpp`、`src/test/voice_core_test.cpp`  
**更新说明**: 本报告用于替换旧版以 Rust 语音模块为核心的调研结论。当前 `main` 上的语音实现已经回到纯 C++ / `QmVoice*` 配置体系，旧版 Rust 覆盖率、Rust 架构优劣、`RiVoiceUseRust` 切换等结论不再适合作为现状文档。

## 一、执行摘要

### 1.1 当前结论

1. 当前语音模块是 **纯 C++ 实现**，核心文件为 `voice_component.*`、`voice_core.*`、`voice_utils.*`，没有现行 Rust 运行路径。
2. 模块整体功能仍然完整覆盖了语音聊天的核心链路：设备初始化、网络收发、Opus 编解码、VAD/PTT、空间音频、房间 Token、可见性/名单过滤、说话叠加层。
3. 当前实现的主要风险不在“功能缺失”，而在 **线程安全、平台音频设备行为、协议一致性、端到端验证不足**。
4. 旧版文档中关于 `src/rust-bridge/voice/`、Rust 超越 C++、`RiVoiceUseRust` 双模切换等内容，已经不代表当前分支现状。

### 1.2 当前状态概览

| 维度 | 结论 |
|------|------|
| 运行架构 | 纯 C++ |
| 核心协议 | 自定义 `RV01` UDP 协议 |
| 音频 I/O | SDL2 音频设备 |
| 编解码 | Opus |
| 降噪 | RNNoise（编译启用时） |
| 控制方式 | VAD / PTT / 测试模式 |
| 识别同步 | 通过 `qmclient.cpp` 的 `/qm/*` HTTP 接口做 Qm 标识、在线分布和语音支持标记 |
| 自动化测试 | 以 `voice_utils` 单元测试为主，缺少端到端语音测试 |

## 二、旧报告已过时的部分

下面这些点在旧版《语音聊天功能调研报告》中已经不再准确：

1. **审查范围错误**  
   旧报告以 `src/rust-bridge/voice/` 为主，而当前语音主实现不在该目录。

2. **“Rust 100% 覆盖 C++ 功能”结论过时**  
   当前现状不是 Rust 接管运行，而是纯 C++ 链路。

3. **`RiVoiceUseRust` / 双模式切换结论过时**  
   当前 `voice_component.h` 只有 `CRClientVoice m_Voice;`，没有 Rust 组件成员，也没有运行时切换接口。

4. **配置命名体系已变化**  
   当前配置前缀是 `QmVoice*` / `qm_voice_*`，而不是旧报告中的 `RiVoice*` 主导。

5. **旧版功能覆盖率统计不可直接复用**  
   旧报告统计的是 Rust 相对 C++ 的覆盖程度，不是当前 C++ 模块本身的完备性。

6. **“Rust 明显优于 C++”类结论不再是现状文档应有内容**  
   当前维护对象是 C++ 模块，应该描述其真实行为、风险和边界，而不是继续围绕 Rust 做优劣对比。

7. **旧版 Windows Opus DLL 修复附录不再是当前语音现状报告重点**  
   它属于特定构建环境问题，和当前功能梳理不是同一层面。

8. **旧版提到的某些 Rust 新增能力（AEC、AGC2、MOS）不能再默认视为现有功能**  
   当前 C++ 现状报告应以 `voice_core.cpp`、`voice_utils.cpp` 的真实代码为准。

## 三、当前模块结构与职责

### 3.1 文件边界

| 文件 | 主要职责 |
|------|----------|
| `src/game/client/components/qmclient/voice_component.h/.cpp` | 组件入口、控制台命令注册、OnInit/OnRender/OnShutdown、叠加层导出 |
| `src/game/client/components/qmclient/voice_core.h/.cpp` | 核心语音运行时：设备管理、工作线程、协议收发、Opus、VAD/PTT、空间音频、房间成员状态 |
| `src/game/client/components/qmclient/voice_utils.h/.cpp` | 协议读写辅助、音频数学工具、名单匹配、增益/压缩/3D 声场计算 |
| `src/game/client/components/qmclient/voice_callback.h` | SDL 音频回调相关接口声明 |
| `src/game/client/components/qmclient/qmclient.cpp` | Qm 识别、在线分布、语音支持标记同步（非语音 PCM/UDP 本体） |
| `src/test/voice_core_test.cpp` | `voice_utils` 相关单元测试 |

### 3.2 组件入口

当前组件入口非常直接：

- `CVoiceComponent::OnInit()` 调用 `m_Voice.Init(...)`
- `CVoiceComponent::OnRender()` 在 `qm_voice_enable` 开启时调用 `m_Voice.OnFrame()`
- `CVoiceComponent::RenderOverlay()` 使用 `CVoiceOverlayState` 渲染左侧语音说话列表

这说明当前没有 Rust/C++ 混跑调度层，UI 和组件层只面向 `CRClientVoice`。

## 四、运行时线程模型

### 4.1 主线程

主线程职责主要有：

1. 读取配置和 UI 输入
2. 维护客户端快照（本地 ID、玩家位置、名字、观战状态）
3. 调用 `OnFrame()` 驱动语音系统生命周期
4. 导出叠加层状态给 UI

### 4.2 Worker 线程

`CRClientVoice` 内部维护独立工作线程，核心循环包括：

1. `EnsureAudio()`
2. `ResolveServerAddr()`
3. `ProcessIncoming()`
4. `DecodeJitter()`
5. `UpdateEncoderParams()`
6. `ProcessCapture()`

Worker 线程同时负责：

- 设备重试
- UDP 收发
- 抖动缓冲处理
- Opus 编解码
- PING/保活
- 发话状态更新

### 4.3 SDL 音频回调线程

输出设备通过 SDL 音频回调拉取混音结果，`MixAudio(...)` 会从各 peer 已解码帧中取 PCM 做混音。

### 4.4 线程模型风险

当前实现把主线程、worker 线程和 SDL 回调线程混合在一起运行，这也是该模块风险最集中的原因之一：共享状态多、边界复杂、回归不易靠单元测试覆盖。

## 五、协议与收发流程

### 5.1 协议头

当前语音协议仍使用 `RV01` 魔数，自定义 UDP 包头大致包含：

1. `VOICE_MAGIC`
2. `version`
3. `type` (`AUDIO` / `PING` / `PONG`)
4. `payload_size`
5. `context_hash`
6. `token_hash`
7. `flags`
8. `sender_id`
9. `sequence`
10. `pos_x`
11. `pos_y`

对应读写辅助在 `voice_utils.cpp` 中统一实现。

### 5.2 发送链路

发送侧核心流程：

1. 从主线程快照同步本地位置、本地 client id、观战位置等上下文
2. 根据配置决定 VAD / PTT / 测试模式
3. 在需要时发送 `PING` / keepalive
4. 从 capture 设备取 PCM
5. 做麦克风增益、降噪、高通压缩
6. 根据 VAD/PTT 判断是否发话
7. Opus 编码
8. 封包并通过 UDP 发送

### 5.3 接收链路

接收侧核心流程：

1. 校验来源地址必须等于当前语音服务器地址
2. 校验包头魔数、协议版本和包类型
3. 校验 `context_hash`
4. 按 `token_hash` / group 规则判断是否可听
5. 对音频包进入 peer jitter 队列
6. 在 `DecodeJitter()` 中解码为 PCM 帧
7. SDL 输出回调线程混音播放

### 5.4 房间与识别同步不是同一条链

需要特别说明：

- **UDP 语音聊天本体**：`voice_core.cpp`
- **Qm 标识 / 在线分布 / voice_supported 标记**：`qmclient.cpp`

二者相关但不相同。用户层面会一起感知为“语音系统”，但调试时必须分开看。

## 七、Qm 识别与在线分布同步

### 7.1 当前职责

`qmclient.cpp` 中的识别同步主要用于：

1. 获取 `Qm` 识别 token
2. 上报本地玩家与语音支持状态
3. 拉取 `users.json`
4. 在本服玩家上打 `Qm` 标识和 `voice_supported` 标记
5. 生成“栖梦客户端在线分布”面板数据

### 7.2 路由分工

当前代码和线上行为体现出两个 HTTP 服务面：

1. **默认语音服务（默认 `:9987`）**
   - `Qm` 识别
   - 在线分布
   - 语音支持标记
   - 路径使用 `/qm/*`

2. **中心 HTTP 服务（`:8080`）**
   - 版本
   - 健康检查
   - 在线时长
   - 路径不使用 `/qm/*`

这部分很容易被误改，因此建议持续保留源码注释说明。

## 九、当前风险、现象与修复方向

### 9.1 高风险：多线程共享状态复杂

`CRClientVoice` 在主线程、worker 线程、SDL 回调线程之间共享大量状态，例如：

- peer 队列
- 音频设备句柄
- 说话状态
- 快照缓存
- 输出混音相关状态

这也是历史代码审查文档中反复提到的最大风险面。

**实际影响：**

- 偶发音频状态不同步
- 设备切换后残留旧状态
- 某些平台上出现“UI 显示已开启，但内部链路未真正就绪”

**修复方向：**

1. 把采集、播放、peer 状态拆成更清晰的状态块，减少跨线程共享字段。
2. 为 `OnFrame()`、worker 循环、SDL 回调之间的共享读写补充明确的线程边界注释。
3. 优先把设备句柄、输入输出启停状态改成统一的状态机，而不是散落的布尔量组合。
4. 针对设备切换、断开重连、退出房间补一轮状态回收检查。

### 9.2 高风险：协议一致性回归难以及时发现

语音协议是手写二进制协议，包头字段较多，任何一个字段写错、偏移错位、配置快照与读包逻辑不一致，都可能表现为：

- `ping = -1`
- `rx = 0`
- 看不到房间内其他说话者
- 听不到别人

这类问题如果没有协议级测试，很容易靠运行时现象才暴露。

**实际影响：**

- 用户第一感知是“语音坏了”，但日志表面上只看到 `peers=0`、`rx=0`
- 一处字段改动可能同时影响房间识别、上下文过滤和收发联通

**修复方向：**

1. 为 `RV01` 包头建立固定测试向量，覆盖 `PING`、`PONG`、`AUDIO` 三类包。
2. 为 `context_hash`、`token_hash`、`sender_id` 建立读写一致性测试。
3. 把协议字段顺序和用途写进源码注释，避免后续再次凭印象改包。
4. 在调试日志中把“已发包”“已收包”“被过滤原因”分开打印，降低排查成本。

### 9.3 中风险：设备枚举与平台差异

当前模块高度依赖 SDL 设备枚举和设备名匹配：

- 设备名大小写
- 已断开设备回显
- 默认设备解析
- 平台权限 / 后端差异

尤其在 macOS 上，输入设备不可用、系统权限、默认设备行为都可能直接表现为 `cap=0`。

**实际影响：**

- 用户会看到语音开关已经打开，但自己听不到、别人也听不到
- 有线耳机、蓝牙耳机、系统默认设备切换后，行为可能和 UI 预期不一致

**修复方向：**

1. 明确区分“语音已启用”“输入设备已打开”“输出设备已打开”“网络已联通”四个状态。
2. 在设备初始化失败时，把失败原因直接暴露到 UI 或调试面板，而不是只留在日志里。
3. 为 macOS 补一份最小排查清单：麦克风权限、默认输入源、SDL 枚举名、是否拿到 capture。
4. 设备下拉选择后，增加一次即时重建和结果反馈，避免用户误以为已经切换成功。

### 9.4 中风险：Qm 识别同步与语音服务地址耦合

识别同步 URL 不是固定的中心域名，而是从当前语音服务器地址推导出来。这样做有复用性，但也意味着：

- 改动语音服务器时会一起影响 Qm 标识和在线分布
- 路由前缀和端口职责容易被误改

**实际影响：**

- 一次看似只改“显示标识”的调整，可能顺带把语音配套识别链路改坏
- `/qm/*` 路由与 `:8080` 中心服务职责混淆时，最容易出现“Qm 标识没了”和“语音也异常”的连锁误判

**修复方向：**

1. 在 `qmclient.cpp` 中保留路径职责注释，明确 `:9987` 与 `:8080` 的分工。
2. 将 Qm 标识同步、在线分布同步、语音联通状态打印拆成独立日志项。
3. 补一条联调说明：改动 `/qm/*` 前必须同时验证标识、`users.json`、语音支持标记。

### 9.5 中风险：端到端验证不足

当前模块功能很多，但自动化验证主要停留在 `voice_utils`，无法覆盖：

- 真实房间联通
- 双客户端通话
- 观战/距离/群组规则
- 实际设备状态机

**实际影响：**

- 很多问题只能靠真实双开、真设备、真房间去碰出来
- 单元测试通过并不代表玩家真的能说、能听、能看到对方

**修复方向：**

1. 增加双客户端联调脚本或手动验证清单。
2. 建立至少一组固定的冒烟用例：本地测试模式、同房间双端互听、Token 不同互斥、观战位置信息同步。
3. 将 `cap`、`out`、`tx`、`rx`、`peers` 等关键运行指标整理成统一调试面板字段。

### 9.6 修复优先级建议

建议按下面顺序处理风险，而不是平均分散推进：

1. **第一优先级：协议一致性与设备状态可观测性**  
   先解决“为什么收不到、为什么采不到、为什么被过滤”这类最难定位的问题。
2. **第二优先级：线程边界收敛**  
   在看得见状态以后，再清理跨线程共享和状态回收问题。
3. **第三优先级：端到端验证沉淀**  
   把已经定位清楚的关键场景沉淀成测试和联调清单。

## 十、功能现状与用户体验评估

### 10.1 当前功能覆盖

从最终用户角度看，当前语音模块已经具备一套相对完整的使用闭环：

1. 可以在设置页启用语音并配置服务器、Token、输入输出设备。
2. 可以在 VAD、PTT、本地测试模式之间切换。
3. 可以看到左侧说话叠加层，感知当前房间内谁在说话。
4. 可以通过 Qm 标识和在线分布感知对方是否可能支持当前语音体系。
5. 可以按距离、全图收听、群组等规则控制可听范围。

### 10.2 当前用户体验优点

当前体验里，真正有价值的点主要有：

1. **配置入口集中**  
   语音相关设置已经集中在 QmClient 的配置界面里，学习成本不高。
2. **房间和身份感知较强**  
   不是单纯“开麦就发”，而是把房间 Token、Qm 标识、在线分布整合进来了。
3. **玩法适配度高**  
   支持空间音频、观战位置、距离衰减，这些都更贴近 DDNet/QmClient 的实际使用场景。
4. **调试信息基础已存在**  
   虽然还不够友好，但 `voice:` 日志与状态字段已经为继续打磨提供了基础。

### 10.3 当前用户体验痛点

从这轮排查过程来看，当前体验上的主要痛点不是“功能没有”，而是“用户不知道系统卡在哪一步”：

1. 开启语音后，用户很难分辨是没拿到麦克风、没连上服务端、没进同房间，还是被协议过滤了。
2. 左侧说话列表、Qm 标识、实际语音联通不是同一个子系统，用户体感上却会把它们视为一件事。
3. 设备切换后的反馈不够直接，尤其在 macOS 和耳机设备场景下，容易误判。
4. 调试字段更偏开发视角，例如 `cap=0`、`rx=0`，普通用户很难据此自助判断问题。

### 10.4 后续用户体验提升方向

如果目标是提升“能不能顺利用起来”，建议优先做下面这些改进：

1. **把状态拆给用户看**  
   在 UI 中明确显示：
   - 麦克风：已打开 / 未打开
   - 扬声器：已打开 / 未打开
   - 服务器：已连接 / 未连接
   - 房间：已匹配 / 未匹配
   - 收发：正在发送 / 正在接收 / 暂无对端
2. **把错误翻译成人话**  
   例如把 `cap=0` 转成“未打开输入设备或无麦克风权限”，把 `peers=0` 转成“当前未发现可通话对端”。
3. **增加一键自检流程**  
   点击后依次检查权限、设备、服务器、Token、收发状态，并输出结论。
4. **优化设备选择反馈**  
   用户选择输入输出设备后，立即提示“已切换成功”或“切换失败，原因是……”
5. **增加首次使用引导**  
   首次开启语音时提示：需要同服、同房间、输入设备正常、对方也支持语音。

### 10.5 后续工程改进方向

如果目标是同时提升“可维护性”和“用户体验”，建议把后续工作拆成三条线：

1. **可观测性线**  
   完善日志、状态面板、错误提示，让问题更快暴露。
2. **稳定性线**  
   收敛线程边界、补协议测试、补设备异常路径处理。
3. **体验线**  
   优化设置页文案、自检流程、设备切换反馈、首次使用引导。

## 十二、最终结论

当前 QmClient 语音模块已经不是旧报告描述的 Rust 主导实现，而是一个 **纯 C++、多线程、SDL+Opus+自定义 UDP 协议** 的完整语音子系统。

它的优势是：

1. 功能面完整
2. 配置项齐全
3. UI/控制台入口成型
4. 已集成 Qm 识别、在线分布与语音支持标记

它的主要问题不是“没有功能”，而是：

1. 线程安全风险高
2. 平台/设备行为复杂
3. 协议回归容易漏检
4. 端到端自动化验证不足

因此，当前更适合把它定义为：

**“功能完整、玩法适配度高，但在可观测性、线程边界和用户自助排障体验上仍需继续打磨的生产模块。”**
