# QmClient 网络卡顿诊断与性能监控完整性报告

## 一、文档定位

本文档记录当前版本中“网络卡顿诊断 + 性能监控面板”的真实实现情况，重点回答三个问题：

1. 现在已经做到了什么程度
2. 对普通用户和开发者分别意味着什么
3. 还缺哪些关键能力，才算真正完成“服务器卡顿 vs 客户端卡顿”的诊断闭环

这份报告不再按“待设计功能”写法展开，而是按现状评估来整理。

---

## 二、当前结论

这块功能已经不是空白方案，而是有完整落地面的现状功能：

- `CQmMonitoring` 已经提供独立的诊断面板
- `CClient` 已经暴露延迟、抖动、预测边距、重发、网络统计等基础信号
- `tc_statusbar` 已经能把下行、上行、抖动、丢包、连接等级、CPU、内存、速率等内容直接显示到状态栏
- 诊断结果已经不是单纯“有问题 / 没问题”，而是四级等级 + 主因分析

对应的核心文件是：

- `src/game/client/components/qmclient/monitoring.h`
- `src/game/client/components/qmclient/monitoring.cpp`
- `src/game/client/components/tclient/statusbar.h`
- `src/game/client/components/tclient/statusbar.cpp`
- `src/engine/client/client.h`
- `src/engine/client/client.cpp`
- `src/engine/shared/config_variables_tclient.h`
- `src/engine/shared/config_variables.h`

## 三、实现现状

### 3.1 诊断面板已经成型

`CQmMonitoring` 现在不是单一数值展示，而是一整套诊断 UI：

- 顶部摘要区：显示连接等级、简要结论、详细解释
- 主图表区：显示延迟、预测值、预测边距、抖动的历史曲线
- 帧率图表区：显示 FPS 与游戏时间边距历史
- 主卡片区：显示延迟、预测值、预测边距、帧率
- 次卡片区：显示抖动、进程下行、进程上行、帧时间、时间回拉

图表面板的入口仍然绑定在 `dbg_graphs` 调试图表开关上，也就是说它已经做出来了，但产品定位仍偏“开发者/高阶玩家工具”，不是默认强曝光的普通 HUD。

### 3.2 网络信号已经分层

当前网络指标并不是只看一个 Ping：

| 指标 | 当前来源 | 现状判断 |
|------|----------|----------|
| 下行延迟 | `SnapshotLatencyMs()` | 已接入 |
| 上行延迟 | `PredictionLatencyMs()` | 已接入 |
| 预测边距 | `PredictionMarginMs()` | 已接入 |
| 抖动 | `PredictionJitterMs()` / `m_AutoMarginLatencyJitterMs` | 已接入 |
| 游戏时间边距 | `GameTimeMarginMs()` | 已接入 |
| 时间回拉 | `QmComputeRollbackMs()` | 已接入 |
| 丢包率 | `PendingResendCount()` + `NETSTATS` 差分 | 已接入，但属于推导值 |
| 下行速率 | `recv_bytes` 差分 | 已接入 |
| 上行速率 | `sent_bytes` 差分 | 已接入 |
| 连接状态 | `ConnectionProblems()` / `IsGameConnectionAlive()` | 已接入 |

这里最重要的变化是：系统已经开始把“卡顿”拆成多种来源，而不只是一个 RTT 数字。

### 3.3 诊断结果不再是二值

`SQmDiagnosticVerdict` 已经把结论拆成：

- 连接等级：`NORMAL` / `ELEVATED` / `SEVERE` / `DISCONNECTED`
- 主因类别：`DOWNSTREAM` / `UPSTREAM` / `JITTER` / `PACKET_LOSS` / `CLIENT_PERFORMANCE`
- 文本摘要与详细解释

这意味着面板已经具备“告诉你为什么卡”的能力，而不只是“告诉你卡了”。

### 3.4 状态栏已经可以做高密度展示

`tc_statusbar` 这条线已经把诊断信息下放到游戏内状态栏，条目包括：

- `u` 下行延迟
- `n` 上行延迟
- `j` 抖动
- `k` 丢包率
- `i` 下行速率
- `o` 上行速率
- `q` 连接等级
- `x` CPU
- `y` 内存

默认 scheme 仍然是 `"ac pf r"`，所以这些高级项虽然已经实现，但不会自动出现在默认布局里，需要用户自己调整状态栏方案。

## 四、UI 结构说明

这块可视化 UI 不是单屏大杂烩，而是按「总览 → 诊断 → 趋势 → 细项」分层组织的。

### 4.1 整体结构

`CQmMonitoring::RenderHud()` 是整个面板的入口，它会先根据屏幕尺寸计算缩放，再把面板切成几块固定语义区域：

1. 外层面板
2. 顶部摘要区
3. 主图表区
4. FPS 图表区
5. 主卡片区
6. 次卡片区

这意味着它不是把所有指标一股脑平铺，而是先给用户一个结论，再逐步展开依据。

### 4.2 顶部摘要区

顶部区域对应 `RenderHeader()`，主要承担“先看结论”的职责：

- 左侧显示连接等级的中文摘要
- 左侧下方显示更具体的原因说明
- 右侧显示一个颜色徽章和等级标签

这一层是整块 UI 最重要的“读秒入口”，用户第一眼应该先看这里，而不是直接钻进图表细节。

### 4.3 主图表区

主图表区对应 `RenderMainGraph()`，它展示四条核心网络曲线：

- 下行延迟
- 上行延迟
- 预测边距
- 抖动

这个区域的作用不是看单点数值，而是看趋势：

- 是持续升高，还是突然抖一下
- 是一路变坏，还是偶发波动
- 是哪条曲线先异常，哪条曲线后跟着坏掉

它更像“网络变化记录器”，不是简单数值卡片。

### 4.4 FPS 图表区

FPS 图表区对应 `RenderFpsGraph()`，它实际上是一个双图结构：

- 上半部分看 FPS
- 下半部分看游戏时间边距

这样做的好处是能把“画面卡”与“同步卡”放在一起看，避免用户只盯着帧率，却忽略了同步链路也在波动。

### 4.5 主卡片区

主卡片区对应 `RenderPrimaryCards()`，是最适合快速扫视的区域：

- 延迟
- 预测值
- 预测边距
- FPS

这一层属于“即时状态摘要”，不需要读图，只要看数字就能知道当前大概处于什么状态。

### 4.6 次卡片区

次卡片区对应 `RenderSecondaryCards()`，放的是更适合排障的补充指标：

- 抖动
- 进程下行
- 进程上行
- 帧时间
- 时间回拉

这一层比主卡片更偏诊断，所以信息密度更高，也更适合高阶用户和开发者。

### 4.7 状态栏是另一条轻量入口

`tc_statusbar` 不属于这个大面板内部，但它是同一套数据源的轻量显示层：

- 大面板适合看全局
- 状态栏适合常驻查看

两者不是重复，而是分工不同。

### 4.8 代码级渲染链路

从实现顺序看，这块 UI 的调用链比较清楚，核心路径可以概括成：

```text
OnRender()
  └─ UpdateSnapshot()
      ├─ UpdateNetworkMetrics()
      ├─ UpdatePerformanceMetrics()
      └─ UpdateDiagnosticVerdict()
          └─ m_Snapshot = Snapshot

RenderHud(View)
  ├─ QmComputeMonitoringUiScale()
  ├─ QmComputeMonitoringBodyLayout()
  ├─ RenderHeader(Header)
  ├─ RenderMainGraph(MainGraph)
  ├─ RenderFpsGraph(FpsGraph)
  ├─ RenderPrimaryCards(PrimaryCards)
  └─ RenderSecondaryCards(SecondaryCards)
```

这一套链路的关键点是：**数据更新和界面渲染是分开的**。  
`UpdateSnapshot()` 先把当前帧要显示的数据整理成 `SQmMonitoringSnapshot`，`RenderHud()` 再只负责消费这个快照去画界面。这样做的好处是，图表、卡片和状态栏都可以共享同一份数据语义，而不是每个控件自己去重复采样。

### 4.9 布局计算方式

布局不是写死死尺寸，而是按屏幕尺寸和可用高度动态拆分：

- `QmComputeMonitoringUiScale()` 根据屏幕宽高算缩放系数
- `QmComputeMonitoringBodyLayout()` 根据内容高度决定图表和卡片各占多少空间
- 当空间充足时，优先保留主图表和 FPS 图表的推荐高度
- 当空间不足时，按最小可用高度压缩，避免整个面板被挤坏

这意味着这块 UI 不是单纯“放个框”，而是带有自己的自适应布局策略。

### 4.10 绘制原语与辅助函数

面板内部还有一组比较重要的绘制辅助函数：

- `DrawCard()`：画卡片底色和圆角外框
- `DrawGraphGrid()`：给图表画网格线，帮助用户判断变化幅度
- `DrawZeroAxis()`：给正负波动图画中线
- `DrawPeakLabel()`：在峰值附近贴标签，方便定位异常尖点
- `DrawSignedPeakLabel()`：用于带正负值的图，例如时间回拉
- `FormatMetricValue()`：格式化普通数值
- `FormatRateValue()`：格式化 KiB/s、MiB/s 这类速率值

这些函数本身不是业务逻辑，但它们决定了这个面板是不是“可读”。  
如果没有这些辅助层，图表会变成一堆没有提示的线段，开发者也很难快速看出峰值和趋势。

### 4.11 状态栏的实现结构

`tc_statusbar` 不是简单的字符串拼接，它也是一套小型的可配置渲染系统：

- `CStatusItem` 负责描述一个条目，包括渲染回调、宽度计算回调、字母编码、名称和说明
- `m_StatusItemTypes` 保存所有可用条目的模板集合
- `m_StatusBarItems` 保存当前启用顺序，默认方案只放少量常用项
- `ApplyStatusBarScheme()` 根据配置字符串把字母编码解析成实际条目
- `UpdateStatusBarScheme()` 会在方案变化时重建布局
- 每个条目最终都走自己的 `XXXWidth()` + `XXXRender()` 组合

这套设计的价值在于：

1. 条目可以按字母编码自由拼装
2. 宽度和渲染是分开的，便于适配不同内容长度
3. 新条目只要补一个 `CStatusItem` 和对应渲染函数，就能接入现有框架

所以从结构上看，状态栏并不是“一个打印一排字”的组件，而是一个轻量的模块化仪表条。

## 五、用户体验评估

### 4.1 对普通用户

普通用户现在能感受到的提升主要有两点：

- 卡顿不再只是“Ping 高”
- 连接状态有了更明确的等级和中文解释

但从产品体验看，这块仍然偏工具型：

- 入口依赖 `dbg_graphs` 或状态栏自定义
- 默认界面不会主动把这套诊断信息推到用户眼前
- `tc_statusbar` 默认 scheme 仍比较保守，诊断信息对新用户不够显眼
- 状态栏里的术语对普通用户不够友好，比如“下行 / 上行 / 抖动 / 预测边距 / 时间回拉”都属于内部诊断语言，第一眼很难理解
- 这些词虽然准确，但没有做面向用户的同义解释或场景化提示，导致“有数据”不等于“看得懂数据”

也就是说，功能是有了，但“用户一眼就能发现并理解”的程度还不够。

### 4.2 对高阶玩家

对高阶玩家来说，这套实现已经很实用：

- 能同时看下行、上行、抖动、丢包、速率
- 能通过历史图表看波动趋势，而不是只看单个瞬时值
- 能用连接等级快速判断当前链路风险
- 能把状态栏当成轻量监控面板使用

这类用户最需要的是“判断依据”，而这套面板已经开始提供判断依据了。

### 4.3 对故障排查场景

如果场景是“到底是服务器卡，还是客户端卡”，当前实现已经比原来强很多，但还没有到完全自动定责的程度：

- 有多个可用信号
- 有初步主因分类
- 但很多结论仍是基于客户端观测值的推导，不是服务器权威指标

所以它更像“高质量的诊断助手”，还不是“最终裁判”。
这里也要明确边界：客户端只能完善客户端侧的观测、推断和展示，公共服务器的服务端实现我们改不了；只有私服或可控服务端场景，才可以进一步补服务端侧指标。

## 六、小白化说明

如果不看技术名词，这套监测可以理解成一份「网络和性能体检表」：

| 名称 | 小白理解 | 看到它变差时通常意味着什么 |
|------|----------|----------------------------|
| 下行延迟 | 服务器回消息到你电脑的速度 | 服务器到你这边的链路变慢了，画面和同步会更拖 |
| 上行延迟 | 你按键、操作发给服务器的速度 | 你的操作反馈变慢了，打钩、起跳、切换会更“沉” |
| 抖动 | 延迟忽快忽慢的程度 | 不是单纯慢，而是不稳定，容易一会儿正常一会儿卡 |
| 丢包率 | 发出去或收回来的数据包有多少没顺利到达 | 网络质量在抖，容易出现卡顿、瞬移、补帧感 |
| 连接等级 | 当前网络健康程度的总分 | 等级越差，说明综合体验越危险 |
| 预测边距 | 客户端为了让画面顺一点，提前留的缓冲 | 缓冲越少，越容易跟不上服务器节奏 |
| 时间回拉 | 客户端发现自己猜快了，被迫往回修正 | 常见于同步压力大时，体感像“被拉回去” |
| 帧率（FPS） | 画面每秒刷新的次数 | 越低越卡，和网络卡不同，主要是本机渲染压力 |
| CPU | 客户端程序占用的计算资源 | 太高时，客户端自己也会拖慢 |
| 内存 | 客户端程序占用的内存空间 | 太高时，可能会带来卡顿或切换不顺 |

换句话说，这些指标不是为了让玩家背术语，而是为了回答三件最重要的事：

1. 是网络慢，还是电脑自己慢
2. 是整体慢，还是一会儿好一会儿坏
3. 是服务器回消息慢，还是我发操作慢

如果把它当成普通玩家能看懂的版本，这份面板更接近「健康状态灯」而不是「工程师调试台」。

## 七、开发者体验评估

### 7.1 结构化程度已经不错

从开发视角看，这块功能的架构是清晰的：

- `CClient` 负责提供底层信号
- `CQmMonitoring` 负责采样、聚合、诊断、渲染
- `tc_statusbar` 负责把信号压缩到高密度展示

这让后续扩展比较顺手，不需要在 UI 层直接硬读网络内部状态。

### 7.2 对调试和迭代友好

当前实现对开发者比较友好的地方是：

- 所有核心指标都收拢到 `SQmMonitoringSnapshot`
- 历史数据有固定容量缓存，方便加图表和峰值标注
- 面板同时包含实时值和历史趋势
- `RenderDebug` 与 `QmMonitoring` 共享数据源思路，减少重复计算

如果后续要补指标，只要继续扩展快照结构和渲染卡片，扩展成本不会太高。

### 7.3 仍然有几处开发债

当前实现还保留着几个典型的“够用但不够硬”的地方：

- 丢包率是推导值，不是底层真实丢包计数
- 连接等级阈值是固定写死的，缺少配置化或自适应策略
- GPU 目前没有作为持续监控指标进入面板
- 帧时间波动还没有形成更明确的 spike 诊断项
- 面板文案有一部分是直接写死的，后续如果继续做国际化，需要再收一遍

这类问题不会马上阻断使用，但会影响后续把它做成“产品级监控面板”。

## 八、完整性判断

如果按“已经实现了多少主体能力”来判断，这个功能已经完成了大头：

- 数据采集链路：基本完成
- 诊断规则链路：基本完成
- 图表与卡片展示：基本完成
- 状态栏高密度展示：基本完成

如果按“是否真正完成服务器卡顿 vs 客户端卡顿的最终诊断”来判断，还是差最后一段：

- 还缺少更权威的服务器侧性能信号
- 还缺少更明确的丢包 / 抖动底层语义
- 还缺少默认用户路径上的强曝光入口
- 还缺少更统一的 GPU / CPU / 网络三位一体监控口径

所以这块功能可以评为：

- **实现完整度：高**
- **用户可发现性：中**
- **诊断权威性：中**
- **开发扩展性：高**

## 九、最关键的未完成项

### 9.1 服务器卡顿 vs 客户端卡顿的权威区分还不够

现在的“主因分析”已经能给出方向，但它更多是客户端基于观测信号的判断，不是服务器侧真值。
这不是当前实现漏做，而是客户端权限边界决定的结果。公共服务器无法由客户端去补服务端真值，只有私服环境才可能做更完整的服务端联动诊断。

### 9.2 下行 / 上行延迟还属于语义拆分，不是完全独立测量

当前展示的是 `SnapshotLatencyMs()` 和 `PredictionLatencyMs()`，它们已经足够有用，但本质上还是客户端侧链路推导，不是完整的一对一单向测量。

### 9.3 丢包率和抖动还偏启发式

现在已经能展示，但计算方式仍是基于重发与间接统计，不是网络层原始计数。

### 9.4 GPU 没有进入持续监控面板

当前面板里有 CPU 和内存，但没有真正意义上的 GPU 使用率、显存占用或渲染压力分解。

### 9.5 默认入口偏弱

`dbg_graphs` + 自定义状态栏方案的组合更适合开发者，不适合作为“普通用户默认就能看懂”的产品入口。

### 9.6 状态栏语义偏陌生

`tc_statusbar` 当前最大的问题不是缺指标，而是指标名称太偏内部实现：

- “下行 / 上行”对网络工程语境是清楚的，但对多数玩家来说不直观
- “抖动”比“延迟波动”更专业，普通用户不一定会把它和卡顿联系起来
- “预测边距”“时间回拉”更像诊断术语，不像可直接消费的 UI 文案

这会带来一个很典型的体验问题：用户即使把状态栏开出来，也可能只是看到一串数字和陌生名词，仍然不知道问题到底来自网络、服务器还是客户端。

### 9.7 还适合继续补充的指标

如果继续完善，这套监测里还有一些**只靠客户端就能继续补强**的指标，比较适合加进去：

| 指标 | 适合原因 | 更适合给谁看 |
|------|----------|------------|
| 帧时间峰值 / 卡顿尖峰 | 现在只有平均帧时间，补上峰值后更容易看出“突然一卡” | 普通用户 + 开发者 |
| 低 FPS / FPS 波动 | 只看平均 FPS 不够，波动大才更容易体感卡顿 | 普通用户 |
| 预测压力 | 已经有现成计算逻辑，能说明客户端预测是不是已经吃紧 | 开发者 |
| 游戏时间回拉次数 / 比例 | 比单次回拉更能看出当前同步压力是不是持续存在 | 开发者 |
| 连接异常持续时长 | 比单纯 `ConnectionProblems` 更直观，能看出问题是偶发还是持续 | 普通用户 |
| 收发速率峰值 / 均值 | 现在已有上下行速率，但再补峰值和平均值会更容易看趋势 | 开发者 |
| GPU 使用信息 | 现在只有 CPU 和内存，GPU 信息更能说明是不是本机渲染压力 | 开发者 |
| 显存 / 纹理 / 缓冲区占用 | `GetGpuInfoString()` 已经能拿到这些信息，适合进一步做持续监控 | 开发者 |
| Ping 来源标记（实测 / 估算） | 让用户知道当前看到的数值到底是网络实测还是浏览器估算 | 普通用户 |

这里的优先级建议是：

1. 先补「帧时间峰值」「低 FPS / FPS 波动」「连接异常持续时长」
2. 再补「预测压力」「游戏时间回拉次数 / 比例」「收发速率峰值 / 均值」
3. 最后再补「GPU / 显存 / 纹理 / 缓冲区占用」这类偏开发者侧的指标

这样可以先把“用户能不能快速看懂”这件事做好，再继续往更细的性能诊断扩展。

---

## 十、建议的后续演进方向

如果继续完善，这块最值得做的是三件事：

1. 把“诊断结论”做成更明确的用户入口，不只依赖调试图表
2. 把丢包、抖动、延迟的语义继续收敛成统一的监控协议层
3. 补上 GPU / 渲染压力的持续指标，让性能面板真正覆盖 CPU、GPU、内存和网络

这样这块功能才会从“能看”升级成“好看、好懂、好排障”。
